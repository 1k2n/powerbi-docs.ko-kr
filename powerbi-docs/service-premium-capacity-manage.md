# <a name="managing-premium-capacities"></a>프리미엄 용량 관리

Power BI 프리미엄 관리는 프리미엄 용량을 만들고, 관리 및 모니터링합니다.

## <a name="creating-and-managing-capacities"></a>용량 만들기 및 관리

Power BI 관리 포털의 **용량 설정** 페이지에는 구매한 v 코어 및 사용 가능한 프리미엄 용량 수가 표시 됩니다. 페이지에서 Office 365 전역 관리자 또는 Power BI 서비스 관리자가 사용 가능한 v 코어에서 프리미엄 용량을 만들거나 기존 프리미엄 용량을 수정할 수 있습니다.

프리미엄 용량을 만들때, 관리자는 다음을 정의해야 합니다.

- 용량 이름 (테넌트 내에서 고유)
- 용량 관리자
- 용량 크기
- 데이터 상주에 대한 영역

하나 이상의 용량 관리자가 할당 되어야 합니다. 용량 관리자로 할당된 사용자는 다음 작업을 수행할 수 있습니다.

- 용량에 작업 영역을 할당 합니다.
- 추가 용량 관리자 또는 할당 권한이 있는 (용량에 작업 영역을 할당할 수 있도록) 사용자를 추가하기 위해 사용자 권한을 관리 합니다.
- 페이지를 매긴 보고서 및 데이터 흐름 작업에 대한 최대 메모리 사용량을 구성하기 위해 워크로드를 관리 합니다.
- 시스템 오버 로드로 인해 모든 작업을 다시 설정하고 용량을 다시 시작 합니다.

용량 관리자는 작업 영역 사용 권한에서 명시적으로 할당 하지 않으면 작업 영역 콘텐츠를 액세스할 수 없습니다. 또한 사용 메트릭, 감사 로그 또는 테넌트 설정과 같은 모든 Power BI 관리자 영역에 액세스할 수 없습니다(명시적으로 할당하지 않는한). 중요한 점은 용량 관리자는 새 용량을 만들거나 기존 용량을 확장할 수 있는 권한이 없습니다. 관리자는 할당된 용량만 보고 관리할 수 있도록 용량별로 할당됩니다.

용량 크기는 풀의 사용 가능한 v 코어의 수로 제한 되는 사용 가능한 SKU 옵션 목록에서 선택 됩니다. 하나이상 구매한 SKU에서 조달할 수 있는 풀에서 여러 개의 용량을 만들 수 있습니다. 예를 들어 P3 SKU (32v-코어)는 세 가지 용량을 만드는 데 사용할 수 있습니다: 하나의 P2 (16v-코어), 두 P1 및 (2 x 8 개 v 코어). [프리미엄 용량 최적화](service-premium-capacity-optimize.md) 문서에서 설명한 대로 더 작은 크기의 용량을 만들어 향상된 성능과 확장성을 달성할 수 있습니다. 다음 이미지는 공유 용량에서 여러 작업 영역과 각 앱 작업 영역을 포함하고 있는 5 개의 프리미엄 용량으로(3 x P1 및 2 x P3) 구성된 가상의 Contoso 조직에 대한 설정의 예제를 보여줍니다.

![가상의 Contoso 조직에 대 한 설정의 예제](media/service-premium-capacity-manage/contoso-organization-example.png)

Power BI 테넌트의 홈 지역이 아닌 다중 지역이라고 불리는 지역에 프리미엄 용량을 할당할 수 있습니다. 다중 지역은 사용자의 Power BI 콘텐츠가 상주하는 정의된 지리적 지역 내의 데이터 센터에 대한 관리 제어를 제공합니다. 다중 지역 배포 하는 이유는 일반적으로 성능 및 확장성 때문이 아니라 회사 또는 정부 규정 준수를 위한 것입니다. 보고서 및 대시보드 로드에는 여전히 홈 영역의 메타 데이터에 대한 요청이 포함 됩니다. 자세한 내용은 [Power BI 프리미엄에 대한 다중 지역 지원](service-admin-premium-multi-geo.md)을 참조합니다.

Power BI 서비스 관리자 및 Office 365 전역 관리자는 프리미엄 용량을 수정할 수 있습니다. 특히 다음을 수행할 수 있습니다.

- 강화 또는 규모 축소 리소스로 용량 크기를 변경 합니다.
- 용량 관리자를 추가하거나 제거합니다.
- 할당 권한이 있는 사용자를 추가하거나 제거합니다.
- 추가 워크 로드를 추가하거나 제거합니다.
- 지역을 변경합니다.

특정 프리미엄 용량에 작업 영역을 할당하려면 할당 권한이 필요 합니다. 전체 조직, 특정 사용자 또는 그룹에 권한을 부여할 수 있습니다.

기본적으로 프리미엄 용량은 Power BI 쿼리 실행과 관련된 워크 로드를 지원 합니다. 또한 프리미엄 용량은 추가 워크 로드를 지원 합니다: **AI (Cognitive Services)** , **페이지를 매긴 보고서**, 및 **데이터 흐름**. 각 워크 로드는 워크 로드에 따라 사용할 수 있는 최대 메모리 (사용 가능한 총 메모리의 백분율)로 구성 해야 합니다. 최대 메모리 할당을 늘리면 새로 고칠 때 처리량 및 호스팅할 수 있는 활성 모델 수에 영향을 미칠 수 있다는 것를 이해 하는 것이 중요합니다. 

메모리는 동적으로 데이터 흐름에 할당되지만 페이지를 매긴 보고서에는 정적으로 할당 됩니다. 최대 메모리를 정적으로 할당 하는 것에 대한 이유는 페이지를 매긴 보고서는 용량이 포함된 보안된 공간 내에서 실행하기 때문입니다. 모델을 로드하기 위한 사용 가능한 메모리가 줄어들기 때문에 페이지를 매긴 보고서 메모리를 설정할 때 주의 해야 합니다. 자세한 내용은 [기본 메모리 설정](service-admin-premium-workloads.md#default-memory-settings)을 참조합니다.

프리미엄 용량 삭제는 가능하며 해당 작업 영역 및 콘텐츠 삭제는 발생 하지 않습니다. 대신 공유 용량으로 할당된 모든 작업 영역이 이동합니다. 다른 지역에서 프리미엄 용량을 만들었을때, 해당 작업 영역은 홈 영역의 공유 용량으로 이동됩니다.

### <a name="assigning-workspaces-to-capacities"></a>용량에 작업 영역 할당

작업 영역은 Power BI 관리 포털에서 또는 앱 작업 영역을 위해서는 **작업 영역** 창에서 프리미엄 용량에 할당할 수 있습니다.

용량 관리자 뿐만 아니라 Office 365 전역 관리자 또는 Power BI 서비스 관리자가 Power BI 관리 포털에서 작업 영역을 대량 할당할 수 있습니다. 대량 할당은 다음에 적용할 수 있습니다.

- **사용자별 작업 영역** -개인 작업 영역을 포함한 해당 사용자가 소유한 모든 작업 영역을 프리미엄 용량에 할당 됩니다. 이미 다른 프리미엄 용량에 할당된 경우 작업 영역의 다시 할당도 포함 됩니다. 또한 사용자도 작업 영역 할당 권한이 할당 됩니다.

- **특정 작업 영역**
- **전체 조직의 작업 영역** -개인 작업 영역을 포함하여 모든 작업 영역이 프리미엄 용량에 할당 됩니다. 모든 현재 및 미래의 사용자 작업 영역 할당 권한이 할당 됩니다. 이 방법은 권장 되지 않습니다. 더욱 효과적인 방법이 좋습니다.

작업 영역 관리자이면서 할당 권한을 가지고 있는 사용자는 **작업 영역** 창을 사용하여 작업 영역을 프리미엄 용량에 추가할 수 있습니다. 

![작업 영역을 프리미엄 용량에 할당 하려면 작업 영역 창 사용](media/service-premium-capacity-manage/assign-workspace-capacity.png)

작업 영역 관리자는 할당 권한을 요구 하지 않고 용량에서(공유 용량으로) 작업 영역을 제거할 수 있습니다. 전용 용량에서 작업 영역 제거는 공유 용량에 작업 영역을 효과적으로 재배치합니다. 프리미엄 용량에서 작업 영역을 제거하는 것은 부정적인 결과가 발생할 수 있습니다. 예를 들어, 공유 콘텐츠가 Power BI 무료 사용자에게 사용 가능할 수 없게 되거나 공유 용량에 지원되는 허용 한도를 초과하는 경우 예약된 새로 고침이 일시 중단됩니다.

Power BI 서비스에서 프리미엄 용량에 할당된 작업 영역은 작업 영역 이름을 장식하는 다이아몬드 아이콘으로 쉽게 식별 됩니다.

![프리미엄 용량에 할당 하는 작업 영역을 식별 합니다.](media/service-premium-capacity-manage/premium-diamond-icon.png)

## <a name="monitoring-capacities"></a>용량 모니터링

프리미엄 용량 모니터링을 통해 관리자는 용량을 수행 하는 방법을 이해할 수 있습니다. Power BI 관리 포털 또는 **Power BI Premium 용량 메트릭** 앱 (Power BI)을 사용하여 용량을 모니터링할 수 있습니다.

### <a name="power-bi-admin-portal"></a>Power BI 관리 포털

각 용량의 경우, 관리 포털에는 **상태** 탭이 설정된 각 워크 로드 및 용량에 대한 요약 메트릭을 제공 합니다. 메트릭은 지난 7 일 동안 평균을 표시 합니다.  

용량 수준에서, 메트릭은 사용 하도록 설정된 모든 워크 로드의 누적이 됩니다. 다음 메트릭을 제공합니다.

- **CPU 사용률** -용량에 대해 사용 가능한 총 CPU의 백분율로 평균 CPU 사용률을 제공합니다.  
- **메모리 사용량** -사용 가능한 메모리의 합계로 평균 메모리 사용량 (GB)을 제공합니다. 

설정된 각 워크 로드에 대해, 다양한 워크 로드 특정 메트릭뿐만 아니라 CPU 사용률 및 메모리 사용량이 제공 됩니다.  예를 들어, 데이터 흐름 작업에 대한  **총 합**은 각 데이터 흐름에 대한 새로 고침 총 수를 표시하며 **평균 기간**은 데이터 흐름에 대한 새로 고침의 평균 기간을 보여 줍니다.

![포털에서 용량 상태 탭](media/service-premium-capacity-manage/admin-portal-health-dataflows.png)

각 워크 로드의 모든 사용 가능한 메트릭에 대한 자세한 내용은 [관리 포털에서 용량 모니터링](service-admin-premium-monitor-portal.md)을 참조하세요.

Power BI 관리 포털의 모니터링 기능은 핵심 용량 메트릭에 대한 요약을 제공 하도록 설계 되었습니다. 더 상세한 모니터링에 대해서는, **Power BI Premium 용량 메트릭** 앱을 사용하는 것이 좋습니다.

### <a name="power-bi-premium-capacity-metrics-app"></a>Power BI 프리미엄 용량 메트릭 앱

[Power BI Premium 용량 메트릭 앱](https://appsource.microsoft.com/product/power-bi/pbi_pcmm.pbi-premiumcapacitymonitoring?tab=Overview)은 용량 관리자에게 사용 가능한 Power BI 앱이며 다른 Power BI 앱처럼 설치됩니다. 대시보드 및 보고서를 포함합니다.

![Power BI 프리미엄 용량 메트릭 앱](media/service-premium-capacity-manage/capacity-metrics-app.png)

앱을 열 때, 대시보드는 사용자가 용량 관리자인 모든 용량을 통해 집계 보기를 표현 하는 다양한 타일을 표시합니다. 대시보드 레이아웃은 5 개의 주요 섹션을 포함합니다.

- **개요** -앱 버전, 용량 및 작업 영역 수
- **시스템 요약** -메모리 및 CPU 메트릭
- **데이터 세트 요약** -데이터 세트, DQ/LC, 새로 고침 및 쿼리 메트릭 수
- **데이터 흐름 요약** -데이터 흐름 및 데이터 집합 메트릭 수
- **페이지를 매긴 보고서 요약** -새로 고침 및 메트릭 보기

고정된 대시보드 타일의 기본 보고서는 대시보드 타일을 클릭하여 액세스할 수 있습니다. 각 대시보드 섹션에서는 자세한 측면을 제공하고 대화형 필터링을 지원 합니다. 

날짜 범위, 용량, 작업 영역 및 워크 로드(보고서, 데이터 집합, 데이터 흐름)으로 슬라이서를 설정하고, 보고서 페이지를 교차 필터링하기 위해 보고서 시각적 개체의 요소를 선택하여 필터링 합니다. 교차 필터링은 특정 기간, 용량, 작업 영역, 데이터 집합 등을 줄일 수 있는 강력한 기법이며 근본 원인 분석을 수행할 때 매우 유용할 수 있습니다.

앱에서 대시보드 및 보고서 메트릭에 대한 자세한 내용은 [앱을 사용하여 프리미엄 용량 모니터](service-admin-premium-monitor-capacity.md)를 참조하세요.

### <a name="interpreting-metrics"></a>메트릭 해석

메트릭은 리소스 사용량 및 워크 로드 작업에 대한 기준을 이해를 설정 하려면 모니터링 해야 합니다. 용량이 저하되면, 어떤 메트릭을 모니터링해야 하는지, 그리고 당신이 내릴 수 있는 결론을 이해하는 것이 중요합니다.

이상적으로, 쿼리는 보고서 사용자에게 응답성이 뛰어난 환경을 제공하도록 초 이내에 완료해야 하고 더 높은 쿼리 처리량을 사용 하도록 설정해야 합니다. 시간이 길어지는 새로 고침 등-백그라운드 프로세스가 진행 중일 때는 일반적으로 덜 우려 됩니다.

일반적으로 느린 보고서는 용량 과열을 표시 될 수 있습니다. 보고서 로드에 실패하면, 이것은 용량 과열을 나타냅니다. 어떤 경우든, 근본 원인은 다음을 포함하여 많은 요인 때문일 수 있습니다.

- **실패한 쿼리**는 확실히 메모리 압력을 나타내며 모델을 메모리에 로드할 수 없습니다. Power BI 서비스는 실패 하기 전까지 30 초 동안 모델을 로드 하려고 합니다.

- **과도한 쿼리 대기 시간**은 몇 가지 이유 때문일 수 있습니다.
  - Power BI 서비스가 먼저 모델을 제거한 다음 예비 모델을 로드(메모리 쓰레싱을 나타내는 긴 쿼리 대기 시간이 수반되지 않는 한, 높은 데이터 세트 제거율만으로는 용량 스트레스를 표시하는 것은 아님)
  - 모델 로드 시간(특히 큰 모델을 메모리에 로드 대기)
  - 장기 실행 쿼리
  - 너무 많은 LC\DQ 연결 (용량 한도 초과)
  - CPU 포화
  - 한 페이지에 지나치게 많은 시각적 개체로 복잡한 보고서 디자인(각 시각적 개체는 하나의 쿼리) 

- **장기 쿼리 기간**은 모델 디자인이 최적화 되지 않은, 하나의 용량에 여러 데이터 세트가 활성화 되며 하나의 데이터 세트가 긴 쿼리 시간을 생성 하는 경우에 특히 나타날 수 있습니다. 이것은 용량이 충분하고, 질문 내 데이터 세트가 최적 이하 또는 단지 느리다는 것을 시사합니다. 장기 실행 쿼리는 다른 프로세스에 필요한 리소스에 대한 액세스를 차단할 수 있기 때문에 문제가 될 수 있습니다.
- **긴 새로 고침 대기 시간**은 메모리를 사용하는 여러 활성 모델로 인한 메모리 부족을 나타내거나 문제가 있는 새로 고침이 다른 새로 고침을 차단합니다(병렬 새로 고침 제한 초과).

메트릭을 사용하는 방법에 대한 자세한 설명은 [프리미엄 용량 최적화](service-premium-capacity-optimize.md) 문서에서 설명합니다.
